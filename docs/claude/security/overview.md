## 3. Routing Architecture

The frontend does **not use react-router or any URL-based router**. Navigation is handled entirely via a `ViewState` enum in `App.tsx`.

### ViewState Enum (`src/types.ts`)
```typescript
enum ViewState {
  LANDING
  LOGIN
  SIGNUP
  FORGOT_PASSWORD
  RESET_PASSWORD
  CHECK_EMAIL
  VERIFY_EMAIL
  AUTH_SELECTION
  SKILL_SELECTION
  ASSESSMENT
  ASSESSMENT_RESULT
  CANDIDATE_DASHBOARD
  EMPLOYER_DASHBOARD
}
```

### Navigation Pattern
- `App.tsx` holds `currentView` in `useState`
- Components receive `onNavigate(view: ViewState)` as a prop callback
- All URL changes are ignored; the page is always `/`
- Deep links, bookmarks, and browser back/forward do not work

### Role-Based Routing
After authentication:
```
user.role === 'candidate'  →  ViewState.CANDIDATE_DASHBOARD
user.role === 'employer'   →  ViewState.EMPLOYER_DASHBOARD
```

## 4. State Management

### AuthContext (`src/contexts/AuthContext.tsx`)
Global authentication state. Consumed via `useAuth()` hook.

**State shape:**
```typescript
{
  user: AuthUser | null
  isLoading: boolean
  isAuthenticated: boolean
}
```

**AuthUser interface:**
```typescript
{
  id: string
  email: string
  name: string
  role: 'candidate' | 'employer'
}
```

**Methods:**
- `signUp(email, password, name, role)` — POST `/api/auth/register/`
- `login(email, password)` — POST `/api/auth/login/`
- `googleLogin(idToken)` — POST `/api/auth/google/`
- `verifyEmail(token)` — POST `/api/auth/verify-email/`
- `requestPasswordReset(email)` — POST `/api/auth/password-reset/`
- `resetPassword(token, newPassword)` — POST `/api/auth/password-reset-confirm/`
- `logout()` — POST `/api/auth/logout/`, clears localStorage

**Persistence:** Tokens and user profile stored in `localStorage`:
```
lune_access_token    — JWT access token
lune_refresh_token   — JWT refresh token
lune_user_profile    — Serialized AuthUser object
```

## 5. Authentication Flow

```
Landing
  └─→ Auth Selection
        ├─→ Login Page
        │     ├─ Email/Password  → POST /api/auth/login/
        │     └─ Google OAuth    → POST /api/auth/google/
        └─→ Signup Page
              ├─ Email/Password  → POST /api/auth/register/
              │     └─→ Check Email (verify prompt)
              │           └─→ Verify Email (token in URL hash)
              └─ Google OAuth    → POST /api/auth/google/

On success:
  role === 'candidate'  →  CandidateDashboard
  role === 'employer'   →  EmployerDashboard
```

### Token Refresh
`src/lib/api.ts` Axios response interceptor:
- On `401 Unauthorized`: attempts token refresh via `POST /api/auth/refresh/`
- On success: retries original request with new token
- On failure: clears localStorage and redirects to login

## 6. API Integration Layer

### Axios Instance (`src/lib/api.ts`)
```typescript
baseURL: import.meta.env.VITE_API_URL || 'http://localhost:8000/api'
```

**Request interceptor:** Injects `Authorization: Bearer <token>` from `localStorage`

**Response interceptor:** Handles 401s with token refresh + retry logic

**Methods:** `get()`, `post()`, `put()`, `delete()`

### Vite Dev Proxy
`vite.config.ts` proxies `/api` → `http://localhost:3001` in development.

## 11. AI Integration

### Dual AI Call Pattern
AI is called from **two places**:

**1. Direct browser calls via `geminiService.ts`**
- Uses `@google/genai` SDK with `VITE_GEMINI_API_KEY`
- Handles: assessment generation, evaluation, career recs, interview Q&A
- **The API key is bundled into the client JavaScript** — visible to anyone

**2. Backend-proxied calls via `/api/ai/` endpoints**
- Gemini key stays server-side
- Same capabilities, safer key handling

This split means the same AI features exist in both places with duplicated logic.

### Gemini Usage in Frontend Services
```
geminiService.ts
  → generateAssessment(skill, difficulty, type)
  → evaluateSubmission(code, question, language)
  → getCareerRecommendations(skills, history)
  → generateInterviewQuestion(topic, role, difficulty)
  → evaluateInterviewResponse(question, answer)
  → generateSkillPassport(skills, certifications)
```

## 16. Security Audit

### Critical

#### Gemini API Key Exposed in Browser Bundle
- **Location:** `src/services/geminiService.ts`, `VITE_GEMINI_API_KEY`
- **Issue:** All `VITE_*` env variables are statically embedded in the compiled JS bundle. Anyone can open DevTools → Sources and extract the key.
- **Risk:** The Gemini API key can be stolen and used by anyone to make unlimited AI calls billed to the project's account
- **Fix:** Remove direct Gemini calls from the frontend entirely. Route all AI requests through the backend (`/api/ai/`), which already exists and keeps the key server-side.

#### JWT Tokens in localStorage
- **Location:** `src/contexts/AuthContext.tsx`
- **Issue:** `lune_access_token`, `lune_refresh_token`, and `lune_user_profile` stored in `localStorage`
- **Risk:** Any XSS vulnerability (including in third-party dependencies) can exfiltrate tokens and fully impersonate users
- **Fix:** Backend should issue tokens as `HttpOnly; Secure; SameSite=Strict` cookies. Frontend should not touch token storage at all.

### High

#### XSS Risk from Unescaped AI Content
- **Location:** Components that render AI-generated text
- **Issue:** AI-generated content (assessment questions, feedback, career recommendations) may be injected into the DOM. If rendered via `dangerouslySetInnerHTML` without sanitization, it becomes an XSS vector.
- **Fix:** Audit all components that render AI or user-provided text. Use `DOMPurify` before any `dangerouslySetInnerHTML`. Prefer React's default JSX escaping.

#### No CSRF Protection on State-Changing Requests
- **Issue:** The app uses JWT Bearer tokens (not cookies) for auth, which avoids classic CSRF. However, if the fix for localStorage is applied (moving to cookies), CSRF protection becomes mandatory.
- **Fix:** When migrating to cookie-based auth, ensure the backend's CSRF middleware is active and the frontend sends the CSRF token with every state-changing request.

#### Third-Party Script Risk (MediaPipe CDN)
- **Location:** `proctoringService.ts`, `@mediapipe/face_mesh`
- **Issue:** MediaPipe may load WASM models from external CDN URLs at runtime
- **Risk:** CDN compromise or MITM could inject malicious code with access to webcam feed
- **Fix:** Self-host MediaPipe WASM assets; set a Content-Security-Policy that disallows external script sources

### Medium

#### No Content Security Policy (CSP)
- **Issue:** No CSP header or meta tag is configured
- **Risk:** Increases the blast radius of any XSS vulnerability — allows inline scripts, eval, and external resource loading
- **Fix:** Define a strict CSP header at the hosting layer (Netlify headers, Vercel headers) and via `<meta http-equiv="Content-Security-Policy">`

#### localStorage as Primary Data Store
- **Location:** All service modules (`assessmentHistoryService`, `notificationService`, `gamificationService`, etc.)
- **Issue:** User data is primarily written to `localStorage` and synced to backend asynchronously
- **Risk:**
  - Data loss if localStorage is cleared
  - Data divergence across devices
  - Sensitive data (scores, history) in an XSS-accessible store
- **Fix:** Make the backend the source of truth; use localStorage only as a cache with TTL

#### Insecure Direct Object Reference Risk
- **Location:** Components that pass `user_id` or `pk` values directly in API requests
- **Issue:** If a component uses a user-supplied or easily guessable ID in an API call, a user could access another user's data
- **Fix:** Backend should enforce ownership on all object-level queries (verify `request.user` owns the requested resource)

#### No Subresource Integrity on External Scripts
- **Issue:** Any external scripts (Google OAuth, analytics, etc.) loaded without SRI hashes can be tampered with
- **Fix:** Add `integrity` and `crossorigin` attributes to all `<script>` and `<link>` tags loading external resources

### Low

#### Sensitive Data in Console Logs
- **Issue:** Development-era `console.log` statements may expose tokens, user data, or AI responses in browser console
- **Fix:** Audit and remove all `console.log` calls that reference tokens, passwords, or user PII before production

#### RealtimeContext is a No-Op
- **Location:** `src/contexts/RealtimeContext.tsx`
- **Issue:** Components that subscribe to realtime channels believe they are receiving live updates, but no data ever arrives
- **Risk:** Notifications and live assessment monitoring silently do not work — users may not realize they are missing updates
- **Fix:** Either implement WebSocket connections or clearly document which features require them

#### Missing Security Headers at Hosting Layer
No evidence of the following headers being set:
```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```
